# ClipBridge 架构设计

## 1. 设计原则

本项目虽然参考旧项目，但新实现要遵守以下原则：

- 先做简单闭环，再做复杂扩展
- 先统一后端协议，再逐个接入客户端
- 尽量减少第一次实现时的抽象层数
- 保留清晰的模块边界，方便调试和演示

## 2. 总体架构

系统分成五个主模块：

- `backend/`：统一后端服务，负责认证、数据存储、同步、分享、管理能力
- `android/`：优先联合开发的客户端，用来最早打通真实同步闭环
- `web/`：优先联合开发的管理端，负责管理与展示，不负责系统级剪切板监听
- `windows/`：第二阶段接入，处理桌面端托盘、剪切板监听和打包问题
- `harmony/`：最后接入，尽量复用前面已经稳定的接口和交互规则

## 3. 推荐目录结构

```text
ClipBridge/
├─ backend/
├─ android/
├─ web/
├─ harmony/
├─ windows/
└─ docs/
```

## 4. 后端职责

后端是整个项目的核心，必须先完成。

后端负责：

- 用户注册、登录、令牌刷新、退出登录
- 设备信息管理
- 剪切板历史记录存储
- 同步游标管理
- WebSocket 实时推送
- 文件中转
- 文本/文件分享
- 管理员设置与审批流程
- AI 能力统一接入、模型调用编排、结果回写

## 5. 核心数据流

### 5.1 文本同步流程

1. 设备 A 监听到本地剪切板变化
2. 设备 A 调用后端接口上传文本
3. 后端保存记录并生成新的同步序号
4. 后端通过 WebSocket 推送给同账号下其他在线设备
5. 设备 B 收到推送后写入本地剪切板
6. 设备 B 回传 ACK，更新自己的同步游标
7. 如果设备离线，则在重连后通过补拉接口获取缺失内容

### 5.2 文件中转流程

1. 客户端上传文件到后端
2. 后端写入文件元数据与文件存储路径
3. 用户在其他端查看文件列表
4. 目标端按需下载文件

### 5.3 分享流程

1. 用户创建文本分享或文件分享
2. 后端生成分享记录和访问 token
3. 外部访问者通过公开取件页访问
4. 后端按规则判断是否过期、是否撤销、是否阅后即焚

### 5.4 AI 附加功能流程

AI 功能统一由后端承接，客户端不直接调用第三方模型服务。

1. 客户端提交文本、图片，或者传入已有历史记录 ID
2. 后端校验权限、读取原始内容
3. 后端根据任务类型调用 AI 模型或 OCR 引擎
4. 后端返回结构化结果给客户端
5. 如有需要，后端把标题、标签、摘要、敏感等级等结果写回数据库

这样做的好处是：

- Android、Web、Windows、HarmonyOS 都能复用一套 AI 接口
- 模型密钥只保存在后端
- 更方便做日志、限流、配额和审计

## 6. 数据模型建议

建议保留并简化旧项目中的核心表：

- `users`
- `devices`
- `auth_refresh_tokens`
- `clipboard_items`
- `device_sync_state`
- `file_assets`
- `share_items`
- `quota_requests`
- `bandwidth_requests`
- `admin_requests`
- `system_settings`

第一阶段可以先建最核心的几张表：

- `users`
- `devices`
- `auth_refresh_tokens`
- `clipboard_items`
- `device_sync_state`

后面再逐步扩展：

- `file_assets`
- `share_items`
- 各类申请与系统设置表
- AI 元数据表或 AI 扩展字段

如果后续引入 AI 功能，建议增加以下扩展数据：

- `clipboard_ai_metadata`
  - 记录 AI 生成的标题、摘要、标签、敏感等级
- `ai_job_logs`
  - 记录 AI 请求类型、耗时、状态、失败原因

## 7. 接口分组建议

后端接口建议按功能分组，便于多个客户端共用：

- `/healthz`
- `/v1/auth/*`
- `/v1/account/*`
- `/v1/devices/*`
- `/v1/clipboard/*`
- `/v1/files/*`
- `/v1/shares/*`
- `/v1/public/*`
- `/v1/sync/*`
- `/v1/ws`
- `/v1/admin/*`
- `/v1/ai/*`

这样做的好处是：

- 后端职责清楚
- Web、Android、HarmonyOS、Windows 都能复用同一套接口
- 后续查问题时更容易按模块定位

## 8. 客户端架构建议

### 8.1 Android

Android 是优先联合开发的一部分，先做成最小可用版本：

- 登录页
- 主页面
- 同步服务
- 历史记录页
- 设备页
- 文件页
- 设置页

### 8.2 Web

Web 不做系统级剪切板监听，重点做管理后台：

- Dashboard
- 剪切板历史页面
- 文件中心
- 分享管理
- 设备管理
- 申请记录
- 管理员页面
- AI 文本整理入口
- OCR 结果查看
- AI 标签与语义搜索入口

### 8.3 AI 功能接入建议

AI 功能建议优先接入 Web 和 Android：

- Web 更适合先做 AI 操作面板、历史记录 AI 增强和语义搜索
- Android 更适合先做文本整理、OCR 和敏感提醒
- Windows 和 HarmonyOS 可以在后续直接复用后端 AI 接口

### 8.4 HarmonyOS

HarmonyOS 放在最后接入，尽量沿用 Android 的页面划分与交互规则，减少重复设计成本。

### 8.5 Windows

Windows 放在后端、Android、Web 基本稳定之后接入，原因是它比移动端多出以下复杂点：

- 桌面剪切板监听
- 托盘常驻
- 开机自启
- 安装包打包

## 9. us2 部署建议

### 9.1 部署原则

- 继续使用 `us2`
- 不覆盖旧系统
- 不修改旧网页入口
- 新项目使用独立目录、独立域名、独立服务名

### 9.2 旧系统现状

当前旧系统可见地址为：

- `https://hy-us2.xushuangbo.top:18443/dashboard.html`

旧项目在 `us2` 上已经存在运行中的网页和后端，因此新项目不能直接复用旧部署目录，也不要直接覆盖原有 nginx 配置。

### 9.3 新项目建议部署方式

建议新项目使用以下独立部署方案：

- Web 域名：`clipbridge-us2.xushuangbo.top`
- Web 端口：`18444`
- API 本地监听：`127.0.0.1:18081`
- nginx 独立站点配置：`clipbridge-us2.conf`
- systemd 独立服务：`clipbridge-api.service`
- 部署目录：`/srv/clipbridge/`

推荐目录结构：

```text
/srv/clipbridge/
├─ backend/
├─ web/
├─ data/
├─ logs/
└─ env/
```

好处是：

- 旧系统和新系统可以并存
- 即使新项目部署失败，也不容易影响旧项目
- 后续验收和截图时，能明确区分“旧版”和“重建版”

## 10. 开发策略

1. 先联合推进后端、Android、Web 的最小闭环
2. 再做 Windows
3. 最后补 HarmonyOS

AI 功能作为附加能力，建议在核心闭环稳定后逐步接入，不要阻塞主线。
