# ClipBridge UI 规划

## 1. 文档目的

这份文档单独记录 ClipBridge 的客户端 UI 规划，主要解决下面几个问题：

- Android 和 Web 后续接入时，页面结构不要各写各的
- 登录、退出、失效跳转、服务地址修改等关键流程语义要统一
- 第一轮先做哪些页面、哪些页面先不展开，要提前定清楚
- 后续扩展文件中心、分享、管理员、AI 页面时，导航结构不要返工

这份文档偏“产品骨架 + 页面布局 + 跳转规则”，不替代接口文档，也不替代后端架构文档。

## 2. UI 总体方向

### 2.1 视觉风格

整体风格统一为：

- 轻量工具台
- 稳定、理性、偏生产力工具
- 干净、克制，不做聊天软件感
- 允许适度科技感，但不走夸张赛博风

### 2.2 色彩方向

第一阶段统一采用浅色主题优先，先不把深色主题作为主开发目标。

建议色彩基调：

- 主色：蓝青系
- 页面背景：浅灰白
- 卡片背景：白色或极浅蓝白
- 强调状态：青绿色
- 警告状态：橙色
- 危险状态：红色

建议使用语义色，而不是直接在页面里散落写死颜色值。

### 2.3 插画和资源使用规则

当前项目已经整理好的资源优先放在：

- `assets/brand/`
- `assets/illustrations/`
- `assets/illustrations-webp/`

建议使用方式：

- Android 优先使用 `PNG`
- Web 优先使用 `WebP`
- 品牌图标统一从 `assets/brand/` 读取

## 3. 信息架构总览

客户端页面统一分成两层：

### 3.1 未登录态

- 启动检查页
- 登录 / 注册页
- 服务地址配置区域

### 3.2 已登录态

第一轮先固定下面几个核心页面：

- 首页 / Dashboard
- 历史记录
- 设备管理
- 我的 / 设置

第二轮再逐步挂入：

- 文件中心
- 分享管理
- 申请记录
- 管理员页面
- AI 工具页

这里的重点不是“先把所有入口做出来”，而是先把模块名称和页面职责定住，避免后面导航反复调整。

## 4. 跨端统一规则

### 4.1 名称统一

Android 和 Web 后续都使用同一套模块名称：

- 首页 / Dashboard
- 历史记录
- 设备管理
- 我的 / 设置
- 文件中心
- 分享管理
- 申请记录
- 管理员页面
- AI 工具

如果某端暂时没有这个页面，也不要换一套别名。

### 4.2 状态语义统一

下面这些状态文案和语义必须尽量统一：

- 当前设备
- 在线
- 已下线
- 登录已失效
- 正在刷新登录态
- 服务不可用
- 网络异常
- 暂无数据

### 4.3 行为规则统一

- 登录成功后进入主页面壳
- 退出登录后回认证页
- 服务地址被修改后，必须清理当前登录态并重新登录
- Access Token 失效时，先尝试 Refresh Token 刷新
- Refresh Token 也失效时，统一回登录页
- 危险操作要二次确认

### 4.4 空状态统一

第一轮空状态优先覆盖：

- 暂无历史记录
- 暂无设备
- 服务离线 / 请求失败
- 搜索无结果

空状态优先展示：

1. 插画
2. 一句明确说明
3. 一个推荐动作

不要只留空白页面。

## 5. Android UI 规划

## 5.1 页面清单

Android 第一轮页面先固定为：

- `SplashScreen`
- `AuthScreen`
- `HomeScreen`
- `HistoryScreen`
- `DevicesScreen`
- `SettingsScreen`

第二轮再补：

- `FilesScreen`
- `SharesScreen`
- `AiToolsScreen`

### 5.2 启动流程

Android 启动流程统一如下：

1. 进入 `SplashScreen`
2. 读取本地服务地址和 token
3. 如果没有本地登录态，跳 `AuthScreen`
4. 如果有 access token，先请求 `/v1/account/me`
5. 如果失败，再尝试 `/v1/auth/refresh`
6. refresh 成功则进入主页面壳
7. refresh 失败则清理本地会话并回 `AuthScreen`

这样做的目的是避免出现“看起来像已登录，实际上 token 已失效”的半登录态。

### 5.3 主页面壳布局

Android 第一轮主壳建议使用：

- `Scaffold`
- 顶部 `TopAppBar`
- 底部 `BottomNavigation`

底部导航固定 4 个标签：

- 首页
- 历史
- 设备
- 我的

这样做的原因：

- 第一轮核心能力刚好可以被这 4 个入口承接
- 比侧边抽屉更直接
- 后面同步状态、历史、设备管理都能自然落位

### 5.4 各页面布局建议

#### `AuthScreen`

布局结构建议：

- 顶部品牌区
- 中上部主插画
- 中间登录 / 注册切换区
- 表单区
- 服务地址配置区
- 底部错误提示 / 操作提示

页面重点：

- `platform` 不需要用户选择，客户端固定传 `android`
- `device_name` 默认自动生成，但允许用户修改
- 服务地址不要单独藏到复杂设置页，第一轮直接放在认证页里

建议资源：

- `assets/illustrations/p0/auth-hero.png`

#### `HomeScreen`

布局结构建议：

- 顶部欢迎区
- 当前账号摘要卡片
- 当前设备卡片
- 快捷操作卡片
- 同步状态或调试提示区

第一轮首页重点不是做复杂控制台，而是承接最常用操作：

- 查看当前账号
- 查看当前设备
- 进入历史页
- 进入设备页
- 手动上传文本
- 手动拉取历史

#### `HistoryScreen`

布局结构建议：

- 顶部标题栏
- 刷新 / 搜索入口
- 历史记录列表
- 空状态插画

第一轮先不做复杂筛选，只做：

- 列表展示
- 下拉刷新或按钮刷新
- 点击查看详情

建议资源：

- `assets/illustrations/p0/empty-history.png`

#### `DevicesScreen`

布局结构建议：

- 顶部标题栏
- 当前设备说明卡片
- 设备列表
- 空状态插画

设备卡片展示字段先固定为：

- 设备名
- 平台
- 设备 ID
- 是否当前设备
- 最近在线时间
- 状态

建议资源：

- `assets/illustrations/p0/empty-devices.png`

#### `SettingsScreen`

布局结构建议：

- 账号信息卡片
- 服务地址卡片
- 会话信息卡片
- 退出登录按钮
- 调试信息或版本信息

第一轮先不要把设置页做成“大杂烩”。
它的职责先只聚焦在：

- 看当前账号
- 看当前环境
- 改服务地址
- 退出登录

### 5.5 Android 跳转规则

- `SplashScreen -> AuthScreen`
  条件：未登录或登录已失效

- `SplashScreen -> MainShell`
  条件：`/v1/account/me` 校验通过，或 refresh 成功

- `AuthScreen -> MainShell`
  条件：登录或注册成功

- `SettingsScreen -> AuthScreen`
  条件：用户主动退出登录，或修改服务地址后强制重新登录

- 任意受保护页面 -> `AuthScreen`
  条件：Refresh Token 已失效，且无法恢复登录态

## 6. Web UI 规划

### 6.1 页面清单

Web 第一轮页面先固定为：

- `/login`
- `/dashboard`
- `/history`
- `/devices`
- `/settings`

第二轮再补：

- `/files`
- `/shares`
- `/requests`
- `/admin`
- `/ai`

### 6.2 Web 布局结构

Web 第一轮统一采用管理台布局：

- 左侧侧边栏
- 顶部栏
- 主内容区

不要照搬 Android 的底部导航。
Web 的定位是管理端，信息密度会更高，侧边栏更合适。

### 6.3 各区域职责

#### 左侧侧边栏

第一轮只放当前真正可用的入口：

- Dashboard
- 历史记录
- 设备管理
- 设置

第二轮新增时再挂：

- 文件中心
- 分享管理
- 申请记录
- 管理员页面
- AI 工具

#### 顶部栏

顶部栏建议固定展示：

- 当前服务地址或环境
- 当前登录用户
- 刷新状态或全局加载提示
- 退出登录入口

#### 主内容区

主内容区负责承接页面级内容，不要把导航、全局状态和局部状态混在一起。

### 6.4 各页面布局建议

#### `/login`

布局建议：

- 双栏或居中单卡布局
- 左侧或顶部放主插画
- 右侧或中部放登录 / 注册表单
- 高级区域放服务地址配置

建议资源：

- `assets/illustrations-webp/p0/auth-hero.webp`

#### `/dashboard`

布局建议：

- 顶部 Hero 区
- 当前账号摘要卡片
- 当前设备摘要卡片
- 快捷跳转卡片
- 最近状态或后续扩展占位

第一轮 Dashboard 的重点是：

- 让用户一登录就知道自己是谁
- 当前连的是哪个服务
- 当前设备是什么
- 下一步去哪里

建议资源：

- `assets/illustrations-webp/p1/dashboard-hero.webp`

#### `/history`

布局建议：

- 标题区
- 搜索框或预留搜索位
- 历史记录列表 / 表格
- 空状态区

第一轮先强调“能看历史”，不急着做复杂搜索面板。

建议资源：

- `assets/illustrations-webp/p0/empty-history.webp`

#### `/devices`

布局建议：

- 标题区
- 当前设备说明区
- 设备列表区
- 空状态区

建议把“当前设备”在视觉上明确标出来，避免用户误操作。

建议资源：

- `assets/illustrations-webp/p0/empty-devices.webp`

#### `/settings`

布局建议：

- 当前账号卡片
- 服务地址卡片
- 会话状态卡片
- 退出登录区域

Web 设置页和 Android 设置页职责保持一致，只是布局更适合桌面宽屏。

### 6.5 Web 跳转规则

- 访问受保护路由时，如果没有有效登录态，重定向到 `/login`
- 登录或注册成功后，统一跳到 `/dashboard`
- 修改服务地址后，清理本地会话并跳回 `/login`
- Access Token 失效时先尝试 refresh
- Refresh 失败时，清理本地会话并跳回 `/login`

## 7. 第一轮实现边界

为了防止第一轮页面越做越散，这里把范围固定下来。

### 7.1 Android 第一轮必须有

- 启动检查页
- 登录 / 注册页
- 主页面壳
- 首页
- 历史页
- 设备页
- 设置页

### 7.2 Web 第一轮必须有

- 登录页
- 管理台主壳
- Dashboard
- 历史记录页
- 设备管理页
- 设置页

### 7.3 第一轮先不展开

- 管理员复杂操作页
- AI 详细工具流
- 文件中心复杂交互
- 分享高级管理
- 多条件筛选和复杂搜索语法

## 8. 后续扩展原则

后面继续加页面时，遵守下面几个原则：

- 不轻易新增平行导航层级
- 尽量挂到已有模块下面
- Android 优先保持少量标签页
- Web 优先保持稳定侧边栏
- 新页面命名优先复用本文档已有模块名

如果后续 UI 方案调整，以这份文档为统一更新入口，不再把布局规则散落写进多个文档里。
