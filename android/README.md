# ClipBridge Android

这是当前 Android 端的首版壳子。

目标不是一次把所有功能做完，而是先把下面这些基础能力跑通：

- 认证登录
- 启动校验
- 主壳导航
- 设备页真实接口
- 设置页基础操作

## 页面结构

- `Splash`
  - 负责本地配置读取和登录态校验
- `Auth`
  - 登录、注册、测试连接
- `Main`
  - 底部 4 个主入口：`首页 / 历史 / AI / 设置`
- `设置 -> 设备`
  - 当前已经接真实设备接口

## 提示与反馈规则

Android 端现在按 Web 端同一套思路收口，避免一个页面里同时出现很多种提示方式。

### 1. 不重要的成功，不提示

这类操作成功后，页面内容本身已经变化，没必要再多弹一层提醒：

- 登录成功
- 注册成功
- 启动校验成功
- 设备列表刷新成功
- 同步开关切换成功
- 普通页面切换成功

### 2. 重要但轻量的成功，用短时 toast

这类操作用户通常需要一个“已完成”的确认，但不值得长期占住页面：

- 测试连接成功
- 设备名称修改成功
- 强制下线设备成功
- 退出登录
- 服务地址修改成功
- 其他临时占位入口点击提示

规则：

- 使用系统 `Toast`
- 默认短时显示
- 不额外在页面内重复显示同一条成功消息

### 3. 失败和阻断问题，使用页面级错误提示

这类问题会阻止用户继续操作，不能靠瞬时 toast 带过：

- 表单校验失败
- 登录失败
- 注册失败
- 服务连接失败
- 设备接口请求失败
- 设备不存在或已被移除
- 登录态失效，需要重新登录

规则：

- 统一显示在当前页面顶部或主要内容区
- 不自动消失
- 等待用户下一次操作或页面切换后清理

### 4. 进行中状态，只用按钮文案和禁用态表达

不要额外弹“加载中”提示。

例如：

- `测试中...`
- `提交中...`
- `保存中...`
- `正在刷新...`

### 5. 危险操作必须确认

当前主要是：

- 强制下线设备

确认框只负责二次确认，不负责成功/失败提示。

## 当前实现约定

- ViewModel 负责：
  - 页面状态
  - 接口调用
  - 成功 toast 事件
  - 页面级错误状态
- Composable 负责：
  - 渲染界面
  - 收集 toast 事件并展示
  - 根据状态决定显示错误块、按钮禁用态、列表内容
- 网络层负责：
  - 请求封装
  - `401 -> refresh -> retry`
  - 统一错误消息

## 设备页说明

设备页当前已经接入：

- `GET /v1/devices`
- `PATCH /v1/devices`
- `POST /v1/devices/offline`

当前后端语义要注意：

- “强制下线设备”当前等价于物理删除设备记录
- 所以下线成功后，设备会直接从列表中消失
- 当前“在线”只是后端 `is_active` 展示，不是实时心跳在线
